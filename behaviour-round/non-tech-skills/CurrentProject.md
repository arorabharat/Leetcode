High level design of Me Tab upsell:

My current project is a fairly complex system covering areas such as user interface, customer event handling, and API standardization. The goal is to 
launch a prime upsell on new page and ensure the pieces of making a upsell can be standardise into plug and play compnents so developer efforts can be reduced.

Layer 1 – Widget Rendering (UI layer)
This layer focuses on rendering a widget that nudges customers to buy a Prime plan at a discounted rate for the next billing cycle. The widget is shown only to customers 
who are in the last 30 days of their plan expiry and do not have autopay enabled. The primary objective is to improve customer retention.

Layer 2 – Eligibility Engine
This layer dynamically determines customer eligibility for the upsell offer. It evaluates business rules such as subscription status, expiry timeline, and other signals to decide the best discounted plan to offer.

Layer 3 – Data Aggregation Controller
This layer builds a controller that provides a dynamic data model used to render the widget. It includes information such as savings amount, plan price, discount percentage, 
and other relevant details required by the UI.

Layer 4 – Action & Payment Integration
This layer enables customers to take action on the upsell offer, such as purchasing the subscription through supported payment methods. While the payment API already exists, 
the current integration effort is high. My work involves aligning stakeholders and service owners to standardize API parameters and building an adapter library that transforms request 
objects into the required parameter format. This ensures the same action flow can be reused across multiple widget placements.

Non-technical contribution
Apart from technical work, I am also contributing to standardizing the overall process for building Prime upsell features by having increased collaboration between WW and IN prime PM. 
This helps ensure reusability across different marketplaces and reduces development effort for future implementations.

## Refined summary

Currently, I am working on a project focused on standardization and convergence between India Prime and Worldwide Prime subscription services. Historically, these two teams have operated separately because India has different regulatory requirements and customer needs. Over time, this led to significant divergence, which reduced the speed of delivery. Although India leveraged some foundational services developed by the Worldwide Prime team, conflicting requirements prevented us from adopting many new features introduced globally. This situation led to leadership escalation due to reduced innovation velocity.

To address this, I initiated a project to drive convergence with the Worldwide Prime architecture while still supporting India-specific requirements. I proposed two fundamental tenets and secured buy-in:

1. Every requirement should be treated as a feature, not a marketplace-specific customization.
2. Each feature should be built as a plug-and-play module rather than embedding custom logic across the system.

At a high level, the Prime signup architecture consists of four layers:

* UI layer (including upsell components)
* Controller/data model layer
* Data providers layer (responsible for fetching data from various sources)
* Fulfillment layer

Each of these layers had key differences between India and Worldwide implementations. For example, India requires OTP authentication during subscription, while Worldwide flows allow subscription activation via API call and attempt payment asynchronously. Because of these different flows, Worldwide teams often could not account for India-specific scenarios when developing new features.

To solve this, I proposed a modular architecture where plug-and-play APIs enable marketplace-specific flows such as OTP collection through bank authentication pages. This allows India flows to remain compliant while maintaining compatibility with Worldwide architecture.

Another major challenge was limited collaboration between Worldwide and India Prime product teams. To improve alignment, I initiated a bi-weekly collaboration forum where both product teams review upcoming features together. This ensured early alignment on long-term direction, reducing back-and-forth communication between engineering and product teams.

We also identified the lack of a centralized source of truth for feature configurations across marketplaces. I proposed creating a configuration store to manage feature enablement consistently. Additionally, we shifted from marketplace-driven development to an experimentation-driven model. Instead of implementing features solely for specific marketplaces, each feature is now introduced as an experiment. If successful, the feature can be scaled globally.

As a result of these changes, alignment timelines improved significantly. Previously, alignment between India and Worldwide teams took up to two weeks. Now, most decisions are aligned within one or two meetings, significantly improving delivery speed and collaboration efficiency.
